# Dropped — Backend

_2026-06-14_

The server that makes the core loop real: **drop → walk → reveal**. The mobile
client is complete; this is the source of truth for the 50 m rule, the home for
drop coordinates, and the place moderation happens. No accounts — an anonymous
device id is the whole identity.

Stack: **Node + TypeScript + Fastify + Postgres/PostGIS + Drizzle + Zod**, over
the standard `postgres.js` TCP driver. Hosted on Neon for now; host-agnostic by
design (see §7).

---

## 1. Shape of it

The API speaks the client's domain types. Responses are **supersets** of the
client's `Secret`/`Drop` (same field names, casing, ms-epoch `createdAt`) plus
server-owned fields the screens need (`mood`, `hearts`, `stoodHere`, and the
per-viewer `sealed`/`saved`/`hearted`). The client's existing types still parse.

The types and geo math are **copied** from the client, not shared via a package
(the two are separate repos). The copies live at `src/domain/clientTypes.ts` and
`src/domain/geo.ts`, each headed with a comment naming the client file as the
source of truth. `src/services/mappers.ts` is the only place that builds the wire
shape, so any drift is a compile error.

Conventions, all matching the client:

- **Identity:** `X-Device-Id: <uuid v4>` on every request. Validated as a UUID,
  registered lazily on first contact. `/health` is the only public route.
- **Errors:** `{ "message": string }` always (axios reads `.message`). Some carry
  extra fields, e.g. a failed reveal adds `distanceMeters`.
- **Timestamps:** ms epoch numbers. **Coordinates:** `{ lat, lng }` (WGS-84).

## 2. Data model

PostGIS, standard features only. `drizzle/0000_init.sql` runs
`CREATE EXTENSION IF NOT EXISTS postgis;` then creates:

| Table      | Key                       | Notes |
| ---------- | ------------------------- | ----- |
| `devices`  | `id text` (the X-Device-Id) | `created_at`. The only "account". |
| `drops`    | `id uuid`                 | `device_id`, `body` (1–280, checked), `mood` (`joy\|ache\|trouble\|wonder`), `place_label?`, **`geog geography(Point,4326)`**, `status` (`visible\|hidden\|pending`), `reveal_count`, `stood_here`, `heart_count`, `created_at`. |
| `reveals`  | `(drop_id, device_id)`    | One row per device-reveal. Drives `reveal_count` + the Found trail. |
| `saves`    | `(drop_id, device_id)`    | Bookmarks. Drives the Saved trail + `saved`. |
| `hearts`   | `(drop_id, device_id)`    | "I feel this". Drives `heart_count` + `hearted`. |
| `reports`  | `id uuid`                 | `drop_id`, `device_id`, `reason`. Feeds moderation. |

The drop's location is a **`geography(Point,4326)`** column with a **GiST index**
(`drops_geog_gix`) — that index is what makes "drops within 50 m of a point" fast.
The query is `ST_DWithin`; distances are `ST_Distance`. Stored coordinates are
**snapped to 5 dp (~1 m)** so we never keep the author's exact GPS fix.

```sql
CREATE INDEX drops_geog_gix ON drops USING gist (geog);
```

## 3. Endpoints

All errors are `{ message }`. All `createdAt` are ms epoch. The response object
for a secret (call it `ApiSecret`) is:

```ts
{
  id: string;
  body?: string;            // omitted when sealed
  drop: { id, coordinate: {lat,lng}, placeLabel?, createdAt };
  createdAt: number;
  revealCount?: number;
  mood: 'joy' | 'ache' | 'trouble' | 'wonder';
  hearts: number;
  stoodHere: number;
  sealed: boolean;          // true => body withheld
  saved: boolean;           // this device's relationship
  hearted: boolean;
  distanceMeters?: number;  // present on nearby + reveal
}
```

### `GET /health`
→ `200 { ok: true }`. Pings the DB. Public.

### `GET /devices/me`
→ `200 { deviceId, createdAt, dropsQuotaRemaining }`.
`dropsQuotaRemaining` is `DROP_DAILY_LIMIT` minus drops created in the last 24 h.

### `POST /drops`
Body: `{ body, mood, coordinate: {lat,lng}, placeLabel? }`.
Screened on ingest (§5). → `201 ApiSecret` (unsealed, it's the author's own).
- `422 { message }` if hard-blocked.
- `429 { message }` if the daily quota is exhausted.

### `GET /drops/nearby?lat&lng&radiusMeters?`
Visible drops within `radiusMeters` (default `NEARBY_DEFAULT_RADIUS_M`, capped at
`NEARBY_MAX_RADIUS_M`), nearest first.
→ `200 { secrets: ApiSecret[] }`. Each is **sealed** (`sealed:true`, no `body`)
unless this device has already revealed it. `hidden`/`pending` drops are excluded.

### `POST /drops/:id/reveal`
Body: `{ coordinate: {lat,lng} }` — a **one-shot** position, used once and never
stored. → `200 ApiSecret` (unsealed, body included; counters bumped).
- `403 { message, distanceMeters }` if outside 50 m.
- `404 { message }` if the drop is missing or not visible.

### `POST /drops/:id/save` · `DELETE /drops/:id/save`
→ `200 { saved: boolean }`.

### `POST /drops/:id/heart` · `DELETE /drops/:id/heart`
→ `200 { hearted: boolean, hearts: number }`.

### `POST /drops/:id/report`
Body: `{ reason }`. → `200 { reported: true }`. Past `REPORT_HIDE_THRESHOLD`
distinct reporters the drop flips to `pending` (shadow-removed).

### `GET /drops/trail/{found|saved|dropped}?limit&offset`
→ `200 { secrets: ApiSecret[], total }`. Trail entries are unsealed (the device
has earned or owns them). `found` = revealed by this device, `saved` = bookmarked,
`dropped` = authored.

## 4. The reveal flow (the part that must not break)

The guarantee is that the **server** decides the 50 m rule from the position the
client sends — never from anything the client claims.

1. The route validates `{ coordinate }` (Zod) and resolves `request.deviceId`.
2. `reveal.service` asks the repo for the distance, computed in Postgres:
   `ST_DWithin(geog, ST_MakePoint(lng,lat)::geography, 50)` plus `ST_Distance`.
3. Outside 50 m → `403 { message:'Too far to reveal', distanceMeters }`.
4. Inside → record `(drop_id, device_id)` in `reveals` (idempotent via
   `ON CONFLICT DO NOTHING`); on the first reveal only, bump `reveal_count` and
   `stood_here`. Return the unsealed secret.
5. The one-shot `coordinate` is never written anywhere.

PostGIS is the primary check. The client's haversine is **copied** into
`src/domain/geo.ts` and used as the **test oracle**: `tests/reveal.spec.ts` runs
against the real app + PostGIS and asserts that ~51 m is rejected, ~49 m is
accepted, the boundary agrees with the oracle, reveals are idempotent, and a
far-away position can never be talked into a reveal.

## 5. Moderation pipeline

The product's hard problem ("cruelty gets erased"). v1 is **ingest filter +
report + status**, with the human review queue _designed here_ but not built as
UI yet.

**On ingest** (`POST /drops`), `moderation.service.screen(body)` returns one of:

- **block** — clear slurs, explicit threats of violence, or doxxing PII (phone,
  email, exact street address). Rejected with `422`.
- **flag** — borderline cruelty. The drop is **created but `status:'pending'`**,
  so it never appears in nearby until a human clears it. "Block hard, flag soft."
- **allow** — clean. Stored `visible`.

**Shadow-removal.** `nearby` and `reveal` only ever serve `status='visible'`.
`POST /drops/:id/report` records a report (deduped per device); once
`REPORT_HIDE_THRESHOLD` distinct devices have reported a drop it flips to
`pending` — gone from the map immediately, awaiting review.

**Status state machine.**

```
            ingest: flag            reports ≥ threshold
   (new) ─────────────────► pending ◄──────────────── visible
     │                         │                          ▲
     │ ingest: allow           │ reviewer approves        │ reviewer approves
     └────────────► visible ◄──┘                          │
                       │                                   │
                       │ reviewer hides ──► hidden ────────┘ (reviewer restores)
```

**Review queue (designed, not yet built).** A reviewer surface lists `pending`
drops oldest-first (joined with their report reasons/counts) and offers three
actions that set `status`: **approve** → `visible`, **hide** → `hidden`,
**leave** → stays `pending`. The data model already supports it (the `status`
column and `reports` table); only an authenticated admin route + UI are missing.
The word lists in `moderation.service.ts` are intentionally small and central —
tune them as review data comes in. `tests/moderation.spec.ts` pins the
block/flag/allow decisions.

## 6. Privacy posture (a product promise)

- The user's **own / live location is never stored.** Only **drops** carry
  coordinates.
- A reveal's one-shot position is used for a single distance check, then
  discarded — it is never written to any table.
- Stored drop coordinates are **snapped to ~1 m** (5 dp) so the author's precise
  fix isn't retained. (The reveal radius is 50 m, so this costs nothing in UX.)

## 7. Host portability

The brief's rule: moving off Neon must be a connection-string change, not a
rewrite. How that's kept true:

- One **`DATABASE_URL`** env var, read in `src/config/env.ts`. Switching host =
  swap that one value.
- The standard **`postgres.js`** TCP driver — **not** `@neondatabase/serverless`
  or any Neon-specific HTTP/WebSocket driver. Neon accepts standard TCP.
- **Standard PostGIS only** (`ST_DWithin`, `geography`, GiST). The migration
  creates the extension itself, so it replays on any Postgres.
- No dependency on Neon branching, scale-to-zero, or its pooler quirks.
- Local dev points the same `DATABASE_URL` at Neon (no Docker required); a native
  local Postgres + PostGIS works identically — drop `?sslmode=require` if unused.

## 8. Wiring the client (integration note)

The client's `features/*/api/` modules are still empty. Each call goes through the
configured axios instance in `src/services/api/` (which already attaches
`X-Device-Id` and normalises errors to `ApiError`). Set its `baseURL` to this
server. Then each feature's api module is a thin wrapper:

```ts
// features/drop/api/createDrop.ts
import { api } from '../../../services/api';
import type { ApiSecret } from '../types'; // = backend ApiSecret superset

export const createDrop = (input: {
  body: string;
  mood: 'joy' | 'ache' | 'trouble' | 'wonder';
  coordinate: { lat: number; lng: number };
  placeLabel?: string;
}) => api.post<ApiSecret>('/drops', input).then(r => r.data);

// features/nearby/api/fetchNearby.ts
export const fetchNearby = (lat: number, lng: number, radiusMeters?: number) =>
  api
    .get<{ secrets: ApiSecret[] }>('/drops/nearby', {
      params: { lat, lng, radiusMeters },
    })
    .then(r => r.data.secrets); // each .sealed === true until revealed

// features/reveal/api/revealDrop.ts — one-shot position from services/location
export const revealDrop = (id: string, coordinate: { lat: number; lng: number }) =>
  api.post<ApiSecret>(`/drops/${id}/reveal`, { coordinate }).then(r => r.data);
//  -> 403 { message, distanceMeters } when too far (axios ApiError.status === 403)

// features/reveal/api/{save,heart}.ts
export const saveDrop = (id: string) => api.post(`/drops/${id}/save`).then(r => r.data);
export const heartDrop = (id: string) => api.post(`/drops/${id}/heart`).then(r => r.data);
export const reportDrop = (id: string, reason: string) =>
  api.post(`/drops/${id}/report`, { reason }).then(r => r.data);

// features/trail/api/fetchTrail.ts
export const fetchTrail = (kind: 'found' | 'saved' | 'dropped') =>
  api
    .get<{ secrets: ApiSecret[]; total: number }>(`/drops/trail/${kind}`)
    .then(r => r.data);
```

Because the responses are supersets of the client's `Secret`/`Drop`, the existing
screens read `secret.drop.coordinate`, `secret.createdAt`, `secret.body` (once
unsealed) etc. unchanged; the new fields (`mood`, `hearts`, `stoodHere`, `sealed`,
`saved`, `hearted`, `distanceMeters`) are additive.
